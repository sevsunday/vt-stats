# AGENTS.md

## Project

VT Stats ‚Äî static-site dashboard for Battlezone: Combat Commander match statistics. Python pipeline processes raw protobuf session data into pre-computed JSON; static HTML/JS/CSS dashboard renders it.

## Before Making Any Change

1. Read the relevant rule file(s) from `.cursor/rules/`
2. Follow `DEVELOPER_GUIDE.md` for schema and architecture details
3. Never skip these ‚Äî they prevent regressions and ensure consistency

## Rule Files

| File | Scope | When to read |
|------|-------|-------------|

| `data-schema.mdc` | py, js, json files | Proto schema, damage semantics, pipeline output format |

## Key Conventions

- `scripts/statsgate.proto` is the single source of truth for raw data schema. All docs, rules, and pipeline code must match it. When it changes, everything downstream updates.
- All data processing happens in the Python pipeline (`scripts/process_stats.py`), never in browser JavaScript ‚Äî with one **documented exception**: `js/all-matches-aggregator.js` performs pure summation over `data/processed/match_contributions.json` to produce the All Matches view's aggregate. This exists so the picker's facet filters (player count, duration band, players, role, etc.) can scope the aggregate without per-match round trips. Per-match aggregation (leaderboards, weapon meta, rivalry matrices) still happens in the pipeline; only the cross-match summation moved client-side.
- All dependencies are vendored locally in `vendor/` ‚Äî no CDN usage.
- All colors come from CSS custom properties (`--kb-*`) ‚Äî zero hardcoded colors in HTML or JS. Visual effects use `--vt-*` variables.
- **Geist** (vendored in `vendor/fonts/`) is the project typeface. Geist Sans for body/UI, Geist Mono for stat values.
- All dashboard styles live in CSS files ‚Äî **zero inline `<style>` blocks** in HTML.
- Dashboard uses tabbed navigation with lazy rendering ‚Äî charts render on first tab activation, not on page load.
- Global player filter is client-side only ‚Äî `getFilteredData()` derives filtered views from loaded JSON using the same lazy re-render pipeline as match switching. `weapon_meta` is recomputed from `leaderboard[].weapon_breakdown` when filtering. `kills.by_vehicle` is always unfiltered. No pipeline changes needed.
- Match picker filter is **separate** from the per-match player filter. `pickerState` (in `js/app.js`, persisted in sessionStorage as `vt.picker.filters.v2`) drives which manifest entries the All Matches aggregate covers ‚Äî not what shows inside a single match. Facets: duration band (radio), player count (multi), submitter (multi), full-roster Players (multi) with two orthogonal toggles: Match-mode (`Any of these` vs `All of these`) and Role (`Any role` / `Commander` / `Thug`). Role is derived per-match: a name in `team_leaders` is the match's commander; otherwise (still in `players[]`) it's a thug. When the user is on the All Matches view, a chip toggle debounce-reruns `loadAllMatches()` so the career table / radar / global meta reflect the filtered subset. The All-Matches-view banner above the aggregate surfaces "Aggregate of K of N matches matching the active filters" + a Clear-filters shortcut.
- Career-roster threshold: `js/all-matches-aggregator.js` enforces a hard-coded `MIN_CAREER_MATCHES = 5` after summation ‚Äî `career_stats[]` rows with fewer matches *in the current scope* are dropped, and `global_rivalries[]` is cascade-filtered to the kept names. `global_weapon_meta` and `meta.*` totals are unaffected; per-match views (Player Leaderboard, kill feed, etc.) are unaffected. The threshold is surfaced via `meta.min_career_matches` (read by UI labels ‚Äî never duplicate the constant) and `meta.players_dropped_by_min_matches` (count of pruned rows). Any new cross-match aggregate that surfaces players in the All Matches view must respect the threshold by reading from the post-prune `career_stats[]` (or by checking `keptNames` if it builds in parallel).
- Processed JSON is the source of truth for the browser; the proto is the source of truth for the pipeline.
- Players are identified by Steam64 IDs. Header maps: `s64_to_nick`, `teamnum_to_s64`, `s64_to_teamnum`.
- Faction determined by slot convention: slots 1-5 = Team 1, slots 6-10 = Team 2.
- Session data lives in `data/sessions/<username>/*.binpb.gz`, organized by submitter.
- Movement Profile (0-100 `activity_score` per player, career-aggregated) is derived from `UpdateTick` position samples in the pipeline. Higher = more active / covered more map. See `.cursor/rules/data-schema.mdc` for the formula and band thresholds, `docs/DATA_DICTIONARY.md` for the full positioning JSON schema.
- T-Key Usage (`positioning.players[].metrics.target_lock_pct`, 0-1 ratio) is captured from `PlayerState.has_target` (the T-key activates a tap-to-toggle target mode against the nearest enemy; the boolean reflects whether a target lock is currently active, not key-press state or aim placement). Availability is gated by the match-global `has_target_lock_data` flag (mirrored on `positioning`, `match`, and manifest entries) which is `false` for pre-schema sessions and matches where no player ever activated target mode. Unlike `activity_score`, `target_lock_pct` is absolute (cross-match comparable) and averaged directly in `career_stats[].mean_target_lock_pct`. Rendered on the 8th "T-Key Usage" axis of the Player Performance Radar.
- Match Highlights (`match_data.highlights`, schema_version 2) is a fixed-slate, always-on per-match award catalog (12 cards). Pre-computed in `compute_highlights()`; the dashboard reads `currentData.highlights` and **never** narrows under `filterState` ‚Äî same passthrough contract as `kills.by_vehicle`. Cards self-omit when their data gates fail (positioning / pickups / T-key / snipe gates), so the section gracefully degrades on legacy / low-roster matches. Bumps `match.schema_version` from `1` ‚Üí `2`; pre-v2 matches simply have no `highlights` block (UI hides the section).
- Team Factions (`match.team_factions`, schema_version 3): per-team ISDF / Hadean / Scion label derived from per-slot ODF first-letter votes. Same passthrough contract as `kills.by_vehicle` and `highlights` ‚Äî match-global, always-unfiltered. Pre-v3 matches simply have no block (UI hides the badge).
- Match Winner (`match.winner`, schema_version 3): inferred match outcome from the cleaned kill feed via a toggle model on recycler/factory destructions (`compute_match_winner()` in `scripts/process_stats.py`). Outcomes: `clean_win` / `contested` / `unclear`. Always emitted; `team` / `loser` are `null` for unclear. Same passthrough contract as `team_factions` ‚Äî match-global, always-unfiltered. Renderers read from `currentData.match.winner` rather than the (filterable) `kills.feed` so a player filter can't corrupt the result.
- VTSR (VT Stats Rating, `data/processed/elo_current.json`): pipeline-side combat ELO emitted by `scripts/elo.py`. Anchored at 1500, soft floor at 1000 with 150-pt linear taper. Seven-axis composite (PvP doubled, pickup excluded). Asymmetric K-factor with loss aversion 0.85 and chess-style provisional decay. **Corpus-wide / picker-filter-unaware** ‚Äî picker narrows the displayed roster only, never recomputes ratings against the filtered subset (filter-contract.mdc has the formal contract). v1 ships `Œ± = 0` (Combat-only); Wins ELO is stubbed at the anchor for every player so the blend math runs through unchanged. To bump Œ±, set `ALPHA = 0.55` in `scripts/elo.py` once winner-attestation data lands; no UI changes needed (the dashboard methodology modal already shows the blend equation). Pipeline-emitted, corpus-wide, JS aggregator passes through unchanged. 404-safe in the dashboard (`window.__vtElo === null` ‚Üí VTSR card hidden, career-table Tier/VTSR cells render `‚Äî`). Full algorithm in `DEVELOPER_GUIDE.md` ¬ß13; output schemas in `docs/DATA_DICTIONARY.md` ¬ß11. **v2 (Phase 12)**: per-match comparison switched from lobby-median performance (`P_med`) to opponent-strength-weighted expected performance (`E_i = 2/(1 + 10^((Rbar - R)/S_R)) - 1` with `Rbar` = **median** of opponent ratings). Outcome scale unchanged at `S_O = 2.5`. Empirical effect: top tail compresses (top players plateau where typical `P_i ‚âà E_i`), mid-band lifts (low-rated players in heavyweight lobbies stop bleeding for "average" performances they were never expected to exceed). Wins ELO blend path unchanged. Bumped `ELO_SCHEMA_VERSION` 1 ‚Üí 2 (constants block grew `expected_score_logistic_scale`; per-delta rows now carry an `expected` field alongside `performance`); bumped `PIPELINE_VERSION` 7 ‚Üí 8 (full re-process). Pre-v2 `peak_vtsr` values are no longer comparable. **v2.1 (post-Phase-12 tuning)**: `S_R` widened from 400 (chess-canonical, but over-compressed our small-population corpus into Tiers 3‚Äì4 only) to **800** (calibrated for our continuous P_i scoring + ~25-player league). No schema change; only `PIPELINE_VERSION` bumped 8 ‚Üí 9 to force re-rating. Iterating `S_R` in the future is the standard knob for spread tuning ‚Äî bigger `S_R` = wider spread = top players plateau higher; smaller `S_R` = tighter spread. No JS or modal logic changes needed beyond updating the displayed value.
- Career Highlights (`career_highlights` block on the All Matches aggregate output): client-side counterpart to per-match Match Highlights. 24 cards built by `buildCareerHighlightsInline()` in `js/all-matches-aggregator.js` ‚Äî 12 `career_*` siblings of the per-match catalog plus 12 cross-match-only originals (`the_champion` / `the_veteran` / `the_workhorse` / `the_carry` / `the_anchor` / `isdf_loyalist` / `hadean_loyalist` / `scion_loyalist` / `the_diplomat` / `map_master` / `streak_king` / `the_polymath`). Cards self-omit when their data gates fail (e.g. Sharpshooter requires `total_shots_fired >= 1000`). Renderer is the existing `renderHighlights()` in `js/app.js` with a new `mode` parameter (`'match'` | `'career'`) that selects between MATCH and CAREER COPY/UNITS tables and between the two DOM containers (`#section-highlights` / `#section-career-highlights`).
- Commanders + Meta tabs on the All Matches view: `#all-tab-commanders` + `#all-tab-meta` panes added in this overhaul. Three Commanders cards (leaderboard / head-to-head / faction-picks chart) + six Meta charts (maps stacked bar, faction by team slot dual donut, faction win rate, duration histogram, player count, matches over time). Both panes lazy-render via `registerTabRenderer()` from the aggregator's new `commander_stats` / `faction_stats` / `meta_charts` blocks. Win % cells degrade to em-dash with explanatory tooltips below the 5-determined-match floor. Faction palette tokens (`--kb-faction-i` / `-e` / `-f`) live in `css/vtstats-theme.css` and are shared by the badges, the faction-picks chart, and all Meta faction visualisations.
- KaTeX (vendored at `vendor/katex/`, v0.16.x): real LaTeX rendering for the VTSR formula tooltip on the dashboard and the methodology section in `docs.html`. Loaded synchronously on `index.html` (pre-rendered via `katex.renderToString()` with the HTML cached in a module-local `vtsrTooltipHtmlCache`); auto-render contrib loaded only on `docs.html`. - ODF Browser is the project's fourth standalone page (alongside dashboard / docs / raw browser). It's a pure read-only consumer of `data/odf.min.json` + `data/audio/*.wav`; it does NOT participate in the pipeline cache key, the picker filter contract, or the per-match player filter ‚Äî there's no "filtered ODF view" notion. The dashboard cross-links into it from `js/app.js` `renderKillFeed()` weapon ODF chips via `<a class="vt-odf-link" href="odf/index.html?odf=<basename>" target="_blank">`. `.vt-odf-link` styling lives in `css/vtstats-theme.css` (loaded by the dashboard); all other `.vt-odf-*` styling lives in `css/odf-browser.css` (loaded only by `odf/index.html`). - Pipeline invocation: `python scripts/process_stats.py` is the single entrypoint and runs in **sync-default + incremental** mode. It pulls the sibling `statsgate/` git clone (`--ff-only`), additively mirrors any new `.binpb.gz` files into `data/sessions/` (never deletes, never overwrites ‚Äî local-only submitter folders are preserved), then processes only matches whose source is new or has changed size since the last run. Cache key is `(match.submitter, match.source_file, match.source_size_bytes)` plus a top-level `PIPELINE_VERSION` constant (currently `1`) for code-change invalidation. **Bump `PIPELINE_VERSION` whenever `process_match()` or any helper it transitively calls (positioning, highlights, weapon meta, rivalry matrices, `_extract_contribution`, etc.) changes output semantics** ‚Äî orthogonal to `match.schema_version` (frontend contract). Flags: `--no-sync` (offline, skip git pull + mirror), `--sync-only` (fetch then exit, no processing), `--force` (ignore cache, reprocess all). Missing `statsgate/` is a soft-skip (the manual-drop-only workflow needs no flag); failed `git pull` is a hard-fail with a `--no-sync` hint.

## When Schema Changes

1. Replace `scripts/statsgate.proto` with the new version
2. Follow `.cursor/rules/schema-migration.mdc` checklist
3. Update pipeline, JSON output, JS rendering, `data-schema.mdc`, and `DEVELOPER_GUIDE.md` ‚Äî in that order
4. Regenerate the raw-browser descriptor: `npx pbjs -t json scripts/statsgate.proto > vendor/protobufjs/statsgate.proto.json`
5. Verify the Node/browser decode path still matches the Python pipeline by running `scripts/verify_proto_decode.mjs` (see the script header for setup) before shipping the new schema
