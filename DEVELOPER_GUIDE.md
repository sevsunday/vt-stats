# VT Stats тАФ Developer Guide & Technical Specification

## 1. Project Architecture

VT Stats is a **static-site dashboard** for BattleZone match statistics. There is no backend framework, no build tools (Webpack/Vite), and no server-side rendering. The architecture has two parts:

1. **Python Data Pipeline** тАФ reads raw `.binpb` protobuf data, aggregates stats, and outputs small pre-computed JSON files.
2. **Static Browser Dashboard** тАФ loads the pre-computed JSON and renders interactive charts and tables.

## 2. Data Schema тАФ Protobuf

The raw match data uses Protocol Buffers (protobuf). The canonical schema is at `scripts/statsgate.proto`.

### Top-Level Message: `ClientStatSession`

| Field | Type | Description |
|---|---|---|
| `header` | `StatHeader` | Match metadata |
| `event_stream` | `repeated StatEvent` | Ordered event stream |

### `StatHeader`

| Field # | Field | Type | Description |
|---|---|---|---|
| 1 | `map` | `string` | Map filename (e.g. `vsrragnor.bzn`) |
| 2 | `start_time` | `google.protobuf.Timestamp` | Match start time (UTC) |
| 3 | `author_nickname` | `string` | Recording player's name |
| 4 | `author_steam64` | `uint64` | Recording player's Steam64 ID |
| 5 | `tick_rate` | `uint32` | Simulation tick rate (typically 20) |
| 6 | `teamnum_to_s64` | `map<uint32, uint64>` | Slot тЖТ Steam64 |
| 7 | `teamnum_to_nick` | `map<uint32, string>` | Slot тЖТ Nickname |
| 8 | `team_1` | `repeated int32` | Player slots on Team 1 (typically 1-5) |
| 9 | `team_2` | `repeated int32` | Player slots on Team 2 (typically 6-10) |
| 10 | `active_config_mod` | `string` | Server configuration mod identifier |

### Event Types

#### `BulletInit` (field 1)
A player fires a projectile.

| Field | Type | Description |
|---|---|---|
| `tick` | `uint32` | Game tick |
| `shooter` | `int32` | Player slot who fired |
| `ordnance_odf` | `string` | Weapon ODF identifier |

#### `BulletHit` (field 2)
A projectile connects with a target.

| Field | Type | Description |
|---|---|---|
| `tick` | `uint32` | Game tick |
| `shooter` | `int32` | Player slot who fired |
| `ordnance_odf` | `string` | Weapon ODF identifier |

#### `DamageDealt` (field 3) + `DamageReceived` (field 4)
These **always occur as adjacent pairs** in the event stream. A `DamageDealt` event is always immediately followed by a corresponding `DamageReceived` event.

**DamageDealt:**

| Field | Type | Description |
|---|---|---|
| `tick` | `uint32` | Game tick |
| `shooter` | `int32` | Damage source player slot (0 = non-player entity) |
| `team` | `int32` | Owning player's slot (1-10), 0 = world prop |
| `ordnance_odf` | `string` | Weapon ODF (may be null тЖТ "Unknown") |
| `amount` | `float` | Damage amount |

**DamageReceived:**

| Field | Type | Description |
|---|---|---|
| `tick` | `uint32` | Game tick |
| `victim` | `int32` | Damage target player slot (0 = non-player entity) |
| `team` | `int32` | Owning player's slot (1-10), 0 = world prop |
| `ordnance_odf` | `string` | Weapon ODF (may be null тЖТ "Unknown") |
| `amount` | `float` | Damage amount |

### Critical: Damage Event Semantics

The `team` field is **always the owning player's slot number (1-10)**, not a faction ID. Faction membership is determined by cross-referencing slots against `header.team_1` and `header.team_2`.

**Five damage scenarios:**

| Scenario | shooter | victim | team (dealt) | team (recv) | Attribution |
|---|---|---|---|---|---|
| Player тЖТ Player | >0 | >0 | shooter's slot | victim's slot | Personal dealt + personal received + rivalry matrix |
| Player тЖТ AI/Structure | >0 | 0 | shooter's slot | AI owner's slot | Personal dealt + asset received |
| AI/Structure тЖТ Player | 0 | >0 | AI owner's slot | victim's slot | Asset dealt + personal received |
| AI/Structure тЖТ AI | 0 | 0 | AI owner slot | AI owner slot | Asset dealt + asset received |
| World prop (team=0) | any | any | 0 | varies | **Shooter skipped** тАФ victim side still attributed if DR has team > 0 |

**Key rules:**
- `dd.team == 0` or `dd.amount == 0.0` тЖТ skip **shooter attribution only** (no dealt credit); **victim received damage is still processed** from the paired DamageReceived event
- `dr.team == 0` тЖТ skip victim attribution (world prop victim)
- `shooter > 0` тЖТ personal dealt credit to that player slot
- `shooter == 0` тЖТ asset dealt credit to `team` (the owning slot)
- `victim > 0` тЖТ personal received credit to that player slot
- `victim == 0` тЖТ asset received credit to `team` (the owning slot)
- Both `shooter > 0` AND `victim > 0` (and shooter not skipped) тЖТ rivalry matrix entry

#### Dormant / Future Event Types

- **`UpdateTick`** (field 5): Per-tick player position and speed data. Currently commented out in the schema but will be enabled in the future. Will produce massive data volumes.

### Faction Resolution and Fallback

Faction (Team 1 vs Team 2) is determined by cross-referencing player slots against `header.team_1` and `header.team_2`. The pipeline includes two layers of fallback:

1. **Header sanity check:** If both "low" slots (1-5) and "high" slots (6-10) exist in `teamnum_to_nick` but one of the team lists is empty, the header is considered incomplete. Both `team_1` and `team_2` are rebuilt from slot convention (1-5 = Team 1, 6-10 = Team 2).
2. **`slot_to_faction()` fallback:** If a slot is not found in either team list, the standard BZ convention is used: slots 1-5 default to faction 1, slots 6-10 default to faction 2.

This ensures all active players are attributed to a faction even when recording clients produce incomplete header data.

### Fullscreen Modal

Every dashboard card has an expand button (`data-expand="section-id"`) that opens the section in a Bootstrap fullscreen modal. For **chart sections**, a fresh `<canvas>` is created in the modal and the chart is re-rendered at full viewport size via registered renderer functions. For **non-chart sections** (tables, HTML content), the card body is cloned into the modal (with `id` attributes stripped to avoid duplicates, and `<canvas>` elements removed to avoid blank cloned charts). Modal charts are destroyed on close.

## 3. Schema Evolution

The protobuf schema is actively developed. Expected changes:

1. **Steam64 IDs on events** тАФ `shooter`/`victim` fields may change from `int32` slot numbers to `uint64` Steam64 IDs, enabling definitive cross-match player tracking.
2. **`UpdateTick` activation** тАФ Position/speed data at 20 TPS will dramatically increase event stream size. The Python pipeline is already designed to skip unknown event types gracefully.
3. **New event types** тАФ Kill, death, vehicle entry/exit, capture events may be added.
4. **Archive format change** тАФ Current: per-map `.zip` files. Future: one master `.gz` containing timestamped `.binpb.gz` files (e.g. `2026-04-05_20-10-54.binpb.gz`). Map name and metadata will come from the protobuf header, not the filename.

**When the schema changes:**
1. Update `scripts/statsgate.proto`
2. Recompile: `python -m grpc_tools.protoc --proto_path=. --python_out=. statsgate.proto`
3. Update `process_stats.py` to handle new event types
4. Re-run the pipeline
5. Update this document and the Cursor rules

## 4. ODF Integration тАФ Weapon Name Resolution

The `odf.min.json` file contains BattleZone Object Definition File data used to resolve raw ordnance ODF strings (e.g. `chaingun_c.odf`) into human-readable weapon names (e.g. `Chain Gun`).

### Resolution Chain

1. `WeaponClass.ordName` тЖТ `WeaponClass.wpnName`
2. `DispenserClass.objectClass` тЖТ `WeaponClass.wpnName`
3. `TargetingGunClass.leaderName` тЖТ `WeaponClass.wpnName`
4. `Vehicle.TorpedoClass.xplBlast` or `Vehicle.GameObjectClass.explosionName` тЖТ parent weapon name (via `DispenserClass.objectClass` тЖТ vehicle тЖТ explosion mapping)
5. **Fallback**: Strip `.odf` extension and display the raw ODF key
6. **Null ordnance**: Display `"Unknown"`

When multiple ODF strings resolve to the same display name, the raw ODF key is appended in parentheses for disambiguation (e.g. `Shell Gun (shellgun_c)`).

## 5. Pre-Computed Output Format

### `matches.json` (manifest)

```json
[
  {
    "id": "2026-04-10T02-31-36",
    "name": "Ragnarok",
    "file": "ragnarok.json",
    "map": "vsrragnor.bzn",
    "date": "2026-04-10T02:31:36.271257+00:00",
    "duration_sec": 883.6,
    "player_count": 10
  }
]
```

### Per-match JSON (e.g. `ragnarok.json`)

| Section | Key Fields |
|---|---|
| `match` | `id`, `map`, `date`, `duration_sec`, `tick_rate`, `player_count`, `config_mod`, `teams` (rosters with slot/name/steam64) |
| `leaderboard[]` | `name`, `slot`, `faction`, `personal` (dealt/received/net/ratio/accuracy/fav_weapon/weapons_used), `assets` (dealt/received), `weapon_breakdown` |
| `faction_totals` | `"1"/"2"` тЖТ player_dealt, asset_dealt, total_dealt, player_received, asset_received, total_received, shots, hits, accuracy |
| `rivalry_matrix` | `{shooterName: {victimName: damage}}` тАФ player-on-player only |
| `top_rivalries[]` | `a`, `b`, `a_to_b`, `b_to_a`, `total` тАФ top 5 bidirectional pairs |
| `weapon_meta[]` | `weapon`, `odf`, `total_damage`, `total_shots`, `total_hits`, `accuracy`, `users` |
| `timeline` | `bucket_seconds` (10), `labels[]`, `by_player`, `by_faction` тАФ damage per bucket |
| `asset_damage` | `by_player`, `by_faction` тАФ AI/structure damage attribution |

### `all_matches.json` (cross-match aggregate)

| Section | Key Fields |
|---|---|
| `meta` | `match_count`, `total_duration_sec`, `maps_played[]`, `date_range` |
| `career_stats[]` | `name`, `matches_played`, `total_dealt`, `total_received`, `overall_accuracy`, `fav_weapon`, `best_match`, `weapon_breakdown` |
| `global_weapon_meta[]` | Same as per-match `weapon_meta` but aggregated |
| `global_rivalries[]` | Top 10 cross-match bidirectional rivalries |

## 6. Styling Standards

### Bootstrap-First

All UI components use Bootstrap 5.3.2 classes. Never write custom CSS for something Bootstrap already provides.

### Theme System

The project uses a CSS variable system prefixed with `--kb-*`. All 44 themes are defined in `themes.css` with both light and dark mode variants.

**Variable categories:**
- `--kb-primary`, `--kb-accent`, `--kb-info`, `--kb-success`, `--kb-warning`, `--kb-danger` тАФ semantic colors
- `--kb-primary-fg`, `--kb-accent-fg`, etc. тАФ foreground colors for solid backgrounds
- `--kb-bg-body`, `--kb-bg-card`, `--kb-bg-subtle` тАФ background levels
- `--kb-text-primary`, `--kb-text-secondary`, `--kb-text-muted` тАФ text hierarchy
- `--kb-border-subtle` тАФ borders

**Rules:**
- **Zero hardcoded colors** in HTML or JS. Always use CSS variables.
- For muted backgrounds, use `*-muted` variants, never `bg-*-opacity-*`.
- For solid backgrounds, pair with `text-*-fg`. For muted backgrounds, use `text-*`.
- Chart.js colors are read from CSS variables via `getComputedStyle()` at render time.

### Theme Switching

`theme.js` exposes `window.KBTheme` with methods for `setTheme()`, `setMode()`, `toggleMode()`. State is persisted in `localStorage` keys `kb-theme` and `kb-mode`.

HTML attributes: `<html data-theme="default" data-mode="dark">`.

## 7. Chart Architecture

Charts use Chart.js 4.4.7 (vendored locally). Key patterns:

1. **Theme-aware colors**: Read `--kb-primary`, `--kb-accent`, etc. via `getComputedStyle()` at render time.
2. **Destroy on switch**: Call `destroyAllCharts()` before rendering new match data.
3. **Player palette**: A fixed 15-color palette for consistent player coloring within a match.
4. **Chart types used**: Line (stacked area for timeline), Bar (horizontal for weapon meta, stacked for player weapons), Doughnut (rivalry cards), Bar (horizontal for weapon accuracy).

## 8. Adding New Matches

1. Place the `.zip` archive (containing `.binpb`) in `data/stats/`.
2. Run: `cd scripts && python process_stats.py`
3. The pipeline automatically discovers all `.zip` files and outputs to `data/processed/`.
4. Refresh the browser.

### Adding New Event Types

1. Update `scripts/statsgate.proto` with the new message definition.
2. Recompile: `python -m grpc_tools.protoc --proto_path=. --python_out=. statsgate.proto`
3. Add processing logic in `process_stats.py` (the main event loop).
4. Add new fields to the output JSON.
5. Update `js/app.js` to render the new data.
6. Update this document and `.cursor/rules/data-schema.mdc`.
