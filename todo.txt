----- 4A. Heatmap calibration tool (BUILD FIRST) -----

Trigger: every map in data/map-registry.json has
image_calibration: null today, so the dashboard falls back to
terrain_bounds for everyone -> most maps are subtly misaligned.
Hand-tuning JSON via file-edit-reload is too slow.

Tasks:
  1. Build calibrate.html (or ?calibrate=1 mode on dashboard).
     - Loads any one match's positioning data + map image.
     - Two draggable corner handles (or 4 sliders) for
       min.x / max.x / min.z / max.z of image_bounds_world.
     - Live re-render of the heatmap on every drag.
     - "Copy JSON" button outputs the calibration block ready
       to paste into data/maps/<key>.json.
  2. build_map_registry.py already preserves the per-map JSON
     calibration across reruns (see lines 265-296). No
     pipeline change needed.

----- 4B. Audit + tune real maps -----

Tasks:
  1. Build tools/heatmap-audit.html: a single-page grid view
     showing every map at thumbnail size with terrain bounds
     outlined and a synthetic heatmap overlaid. One-glance
     answer to "which maps need fixing?".
  2. Hand-calibrate the top N most-played maps using the tool

  3. Look for batch patterns: all {bac}appel maps may need
     the same offset because they were screenshotted with the

----- 4C. Document the workflow -----

In DEVELOPER_GUIDE.md "Map Assets and Overlays":
  - The two-source precedence (image_calibration ->
    terrain_bounds).
  - Why iondriver images often don't match terrain extents
    (their screenshots crop to playable area; engine reports
    full terrain extent).
  - Step-by-step using the calibration tool.
  - Calibration values can be negative (terrain origin is
    map-center for many maps).

Do NOT:
  - Auto-derive image bounds from terrain_bounds. They're
    often genuinely different and any heuristic will be wrong
    as often as right.
  - Crop/resize the underlying images. Calibrate the
    projection, not the asset.
